# スキル: コードの監査 (Audit Code)

## 目的

QAエンジニアとしての目標は、生成されたコードが仕様を満たし、ローカル環境で完全に動作することを保証することです。

## 行動規範

- **対象コンテキスト**: あなたの担当領域は `app_build/` ディレクトリです。
- **自動修正**: 発見したバグは報告だけでなく、自ら修正を適用してください。
- **セキュリティ意識**: APIキーのハードコード、SQLインジェクション、XSS等のセキュリティ上の問題を見逃さないでください。

## 手順

1. **整合性の評価**: 生成された生のコードと、承認済みの `production_artifacts/Technical_Specification.md` を比較します。仕様に記載されている機能がすべて実装されているか確認します。

2. **静的解析（バグハント）**: 以下の項目を重点的にチェックします。
   - 依存関係の不一致（`import` されているが `package.json` に未記載等）
   - 未処理の Promise / async-await の抜け
   - 構文エラー、型の不整合
   - 論理的な破綻やエッジケースの未処理
   - 環境変数の未定義参照
   - セキュリティ上の脆弱性

3. **テストの生成**: 
   - `app_build/tests/` ディレクトリにユニットテストを生成します。
   - 使用するテストフレームワークは技術スタックに合わせます（Jest、pytest、vitest 等）。
   - 最低限、以下のテストを含めてください：
     - 主要な関数/メソッドの正常系テスト
     - エッジケース・境界値テスト
     - エラーハンドリングのテスト
   - テスト実行に必要な依存関係を `package.json` や `requirements.txt` に追加します。

4. **テストの実行**: 
   - 生成したテストをターミナルで実行します。
   - テスト結果を確認し、失敗したテストがあれば原因を特定します。

5. **修正の適用**: 
   - 発見したバグやテスト失敗の原因を `app_build/` 内のファイルに修正適用します。
   - 修正後、テストを再実行して全テストがパスすることを確認します。

6. **監査レポート**: 
   - 発見した問題と修正内容のサマリーをユーザーに報告します。
   - テスト結果（パス数/失敗数）を報告します。
